# 이슈 템플릿 모음 (병렬 개발/검토용)

## 1) Epic 템플릿
```md
## [EPIC] <스레드명>: <목표>

### 배경
<왜 필요한지>

### 범위
- ...

### 제외 범위
- ...

### 산출물
- [ ] 코드
- [ ] API/스키마 문서
- [ ] 테스트
- [ ] 운영 알람/로그

### 수용 기준 (Acceptance Criteria)
1. ...
2. ...
3. ...

### 의존성
- 선행: ...
- 병렬 가능: ...

### 리스크
- ...

### 완료 기준(DoD)
- ...
```

## 5) 스레드별 Epic 초안 (복붙용)

### Epic A
```md
## [EPIC] Thread A: 도메인 모델/워크플로우 계약 확정

### 배경
병렬 개발을 위해 상태머신/스키마/API/이벤트 계약이 먼저 고정되어야 한다.

### 범위
- PostgreSQL 핵심 테이블 설계
- 상태 전이/권한 규칙 설계
- OpenAPI + 이벤트 스키마 확정

### 수용 기준 (Acceptance Criteria)
1. 상태 전이가 문서와 코드에서 일치한다.
2. B~F 스레드가 참조 가능한 계약 문서가 존재한다.
3. 계약 위반 요청이 표준 에러로 반환된다.
```

### Epic B
```md
## [EPIC] Thread B: 데스크/기자 CMS 구축

### 배경
AI 자동화 이전에도 수동 송고/데스킹이 가능한 운영 베이스가 필요하다.

### 범위
- 기자/데스크 RBAC
- 기사 편집/승인/예약/정정
- 리비전 및 감사 로그 조회

### 수용 기준 (Acceptance Criteria)
1. 기자는 승인/발행 기능에 접근할 수 없다.
2. 데스크는 승인/예약/회수를 수행할 수 있다.
3. 발행 후 수정 이력이 누적 저장된다.
```

### Epic C
```md
## [EPIC] Thread C: AI 인제스트/초안 송고 자동화

### 배경
지속 수집-초안 생성 파이프라인이 있어야 AI 기자 운영비 절감 효과가 발생한다.

### 범위
- 수집기/중복제거/생성기/검증기/송고기
- 실패 재시도 + 데드레터 큐

### 수용 기준 (Acceptance Criteria)
1. 지정 소스에서 자동 초안이 `desk_review`로 저장된다.
2. 중복 송고가 멱등성 키로 제어된다.
3. 초안별 근거 링크/모델 버전이 저장된다.
```

### Epic D
```md
## [EPIC] Thread D: 발행 엔진 및 읽기 전용 웹

### 배경
발행된 기사를 고성능 정적 페이지로 제공해야 트래픽/SEO 대응이 가능하다.

### 범위
- Next.js 기사/목록/섹션 페이지
- ISR 재생성 트리거
- SEO/OG/Schema 자동화

### 수용 기준 (Acceptance Criteria)
1. 승인 후 목표 시간 내 기사 반영.
2. 정정/회수 상태가 웹에 정확히 반영.
3. SEO 필수 메타 누락률 기준 충족.
```

### Epic E
```md
## [EPIC] Thread E: 포털 송고 XML/RSS 자동화

### 배경
포털 유입 확보를 위해 표준 규격 피드와 빠른 갱신 반영이 필요하다.

### 범위
- 포털별 피드 생성
- ping/재전송/실패 추적

### 수용 기준 (Acceptance Criteria)
1. 규격 검증 통과.
2. 발행/정정/회수 이벤트 반영 지연 기준 충족.
3. 실패 건 재송신 및 원인 추적 가능.
```

### Epic F
```md
## [EPIC] Thread F: 운영/보안/관측성 체계

### 배경
뉴스 서비스는 장애/오보/보안 사고의 영향이 크므로 운영 안전장치가 필수다.

### 범위
- 모니터링/알림/런북
- 키 관리/권한/감사
- 비용 추적

### 수용 기준 (Acceptance Criteria)
1. 핵심 장애 시나리오에서 탐지와 알림이 동작.
2. 백업/복구 리허설 완료.
3. 비용 임계치 경보가 설정됨.
```
