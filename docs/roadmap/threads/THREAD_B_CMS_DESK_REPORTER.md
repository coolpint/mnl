# Thread B: CMS (데스크/기자)

## 목표
- 인간 기자 송고와 데스크 승인/예약/정정 워크플로우를 수행하는 관리자 시스템을 구축한다.

## 범위
- 역할 기반 접근 제어(RBAC): `reporter`, `desk`, `editor_in_chief`, `ops_admin`, `portal_admin`
- 기사 목록/상세/편집/상태 전환 UI
- 버전 이력(Revision) 조회 및 롤백(선택)
- 대표 이미지 업로드 및 캡션 관리
- 대시보드(대기 큐, 예약 큐, 오류 로그)
- 엠바고/예약발행/예약취소/긴급수정
- 기사 검색/카테고리/기자 페이지 메타 관리

## 구현 트랙 (2026-03-02 업데이트)
- Thread-B CMS 서버 구현은 `apps/cms-wagtail`을 기준으로 진행한다.
- 기존 `apps/cms`(Next 스켈레톤) 의존은 제거하고 Wagtail 단일 트랙으로 운영한다.
- Wagtail 도입에 따른 원본 데이터 경계는 `REQ-20260302-05` master 결정으로 확정한다.
- AI 초안 유입은 `POST /api/v1/newsroom/intake/ai-draft/` 계약(v1)으로 고정하고 master(`REQ-20260302-08`)에서 관리한다.
- Render 배포 시작 커맨드에서 `bootstrap_cms_reporter_user`를 실행해 기자 계정을 `reporter` 권한으로 유지한다.
- `scourt` Teams 리포트는 `import_scourt_reports` 커맨드로 AI 재작성 후 `writing` 초안으로 유입한다.

## 작업 항목

2. 기사 편집기 + 제목 추천 선택 UI

## RBAC 세부 권한 매트릭스 (초안)

| 액션 | reporter | desk | editor_in_chief | ops_admin | portal_admin |
| --- | --- | --- | --- | --- | --- |
| 기사 작성/수정 (`draft`, `writing`, `desk_rework`) | O | O | O | - | - |
| `desk_review` 전환 | O | O | O | - | - |
| 승인/반려 (`desk_review -> approved/desk_rework`) | - | O | O | - | - |
| 예약/예약취소 (`desk_review -> scheduled`, `scheduled -> desk_review`) | - | O | O | - | - |
| 발행 후 수정 승인(`published_updated`) | - | O | O | - | - |
| 긴급 회수/보관 (`retracted`, `archived`) | - | - | O | - | - |
| 운영 장애 대응(실패 큐 조회/재처리) | - | - | O | O | - |
| 포털 송고 큐 조회/재전송 | - | - | O | O | O |

참고: 역할 추가/삭제 또는 승인 게이트 예외는 `master/MASTER_INBOX.md` 사전 승인이 필요하다.

## 상태 전이 UI/검증 기준
- UI는 권한 없는 상태 전이 액션 버튼을 숨기고, URL 직접 접근 시에도 서버가 최종 차단한다.
- 서버 권한 오류는 `403`, 상태 전이 불가(`from -> to` 미허용)는 `409`, 입력 검증 실패는 `422`로 응답한다.
- `scheduled` 상태 입력/표시는 `Asia/Seoul` 기준으로 제공하고 저장은 UTC로 정규화한다.
- `published` 이후에는 `slug` 입력을 readonly 처리하며, API 변경 요청은 거부한다.

## 발행 후 수정 정책 강제 (`L1/L2/L3`)
- 공통 게이트: `published`, `published_updated` 상태 기사 저장 시 수정 레벨 선택 없이는 저장할 수 없다.
- `L1`(경미): 오탈자/캡션/링크 중심 수정만 허용한다. URL은 유지하고 포털 재송고 기본값은 비활성으로 둔다.
- `L2`(중요): 제목/수치/핵심 사실 수정을 허용한다. 수정 고지 입력을 필수로 하고 포털 업데이트 송고를 기본 활성화한다.
- `L3`(중대): 기존 기사 본문 전면교체를 금지한다. "정정/추후/반론 기사 작성" 플로우로 신규 기사 생성 후 원기사 관계 연결로만 처리한다.
- 정책 위반 저장 시 서버는 `422`를 반환하고, UI는 어떤 조건을 충족해야 하는지 명시한다.

## 구현 우선순위 (제안)
1. RBAC 미들웨어 + 라우트 가드 + 액션 권한 훅
2. 기사 편집 화면 상태 액션바(승인/반려/예약/회수)
3. 예약 모달(시간대 고정) + 예약 취소
4. 발행 후 수정 모달(`L1/L2/L3`) + slug 잠금
5. 리비전/감사로그 조회 + 운영 대시보드 큐 연동

## 완료 기준 (DoD)
- 기자는 승인 버튼 접근 불가
- 데스크는 승인/예약/회수/수정요청 가능
- 편집책임자는 긴급 회수/보관 전환 가능
- 상태 변경 시 감사 로그 생성
- 발행 후 수정 시 revision 누적
- L3 수정은 신규 기사 생성 + 원기사 연결로만 처리

## 검토 체크리스트
- 승인 권한 우회 경로가 없는가
- 예약 시 타임존 혼선이 없는가
- 대량 편집 시 데이터 손실이 없는가
- 이미지 업로드 실패/재시도 UX가 안전한가
- 발행 기사의 slug 변경 시도가 차단되는가
