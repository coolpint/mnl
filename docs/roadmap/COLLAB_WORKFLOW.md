# 병렬 개발/동시 검토 워크플로우

## 브랜치 전략
- 기본 브랜치: `main`
- 스레드 브랜치: `codex/thread-a-*`, `codex/thread-b-*` ... `codex/thread-f-*`
- 기능 브랜치 예시: `codex/thread-c-ingestion-rss-adapter`

## PR 분리 원칙
- 한 PR은 한 스레드의 한 책임만 포함
- 스키마 변경 PR과 UI 변경 PR 분리
- 계약(API/이벤트) 변경 PR은 반드시 문서 동반

## 리뷰 할당 규칙
- A 관련 PR: B/C/D/E 담당자 중 1명 이상 필수 리뷰
- B/C/D/E 관련 PR: A 담당자 1명 리뷰 권장
- F 관련 PR: 전 스레드 대표가 알람/로그 영향 검토

## 병렬 진행 시 충돌 최소화
- `schema`, `api`, `events` 폴더를 계약 레이어로 고정
- 계약 레이어 변경은 RFC 태그로 사전 공지
- UI/워커/배포 로직은 계약 레이어를 참조만 하도록 유지

## 데일리 운영
- 데일리 체크 항목:
  - 어제 머지된 계약 변경
  - 오늘 예상 충돌 파일
  - 릴리즈 블로커 여부
- 주간 데모:
  - 스레드별 E2E 3분 데모
  - 실패/예외 시나리오 1개 포함

## 머지 게이트
- 필수:
  - 테스트 통과
  - 문서 갱신
  - 체크리스트 완료
- 조건부:
  - 보안/운영 영향이 크면 F 승인 필수
